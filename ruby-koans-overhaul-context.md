Minha conclusão, em termos pedagógicos, é esta: **Ruby Koans é mais uma ferramenta de leitura ativa e descoberta conceitual do que uma ferramenta realista de prática de engenharia Ruby**. ## O que acontece de verdade no Ruby Koans

O repositório-fonte do Ruby Koans não é exatamente o que o aluno vê. Ele tem arquivos em `src/`, e o `Rakefile` gera os arquivos finais removendo soluções ocultas. O gerador remove trechos entre marcadores `#--` e `#++`, troca formas como `__(valor)` por `__`, e monta os arquivos em `koans/`. ([GitHub][3])

Isso é importante porque o projeto parece ter “soluções” nos arquivos-fonte, mas no material final o aluno recebe lacunas. A lacuna típica não é “implemente uma função em um arquivo de produção”; é “preencha o valor esperado dentro de um teste”. O primeiro exemplo oficial é literalmente trocar `assert false` por `assert true` em `about_asserts.rb`. ([GitHub][1])

A infraestrutura de teste também não é Minitest, RSpec ou Test::Unit. Ela é uma mini-framework própria chamada `Neo`, com métodos próprios como `assert`, `assert_equal`, `assert_raise`, `assert_match` etc. ([GitHub][4]) O runner coleta métodos `test_*`, executa os koans em ordem, captura falhas e para no primeiro ponto de erro. ([GitHub][4]) O `Rakefile` dos koans até faz `require 'rake/testtask'`, mas a task principal apenas executa `ruby path_to_enlightenment.rb`; ele não configura uma suíte real de `Rake::TestTask`/Minitest. ([GitHub][5])

Então, respondendo diretamente: **não, não é um projeto inteiro no estilo Rustlings; sim, a maior parte é baseada em editar assertions/respostas dentro dos próprios testes**.

## Então, ele é “mexer em assertions”?

A resposta mais precisa é: **sim, na maior parte do tempo**.

Mas eu separaria em três níveis.

Primeiro, há os **koans conceituais**, que são a maioria. Neles, o aluno altera o próprio arquivo de teste, geralmente substituindo `__` por um valor, classe, regex ou expressão. O teste não é exatamente uma especificação externa; ele é o material didático. Isso vale para strings, arrays, hashes, métodos, blocos, exceptions, classes, módulos, escopo, message passing e afins.

Segundo, há os **mini-projetos guiados por testes**: triângulo, scoring do Greed, dice set e proxy object. Esses são muito mais fortes pedagogicamente para quem gosta do modelo Rustlings, porque os testes deixam de ser o alvo e passam a ser o guia. O aluno escreve implementação. Esses arquivos são a melhor parte do Ruby Koans para “aprender fazendo”.

Terceiro, há o **extra credit**, que é livre demais: pede para criar um jogo Greed, mas não fornece a mesma estrutura de feedback. Isso pode ser bom para autonomia, mas é pior para aprendizado incremental.

## O contraste com Rustlings

Rustlings é diferente em uma dimensão fundamental: ele usa o **toolchain real da linguagem** como parte do ensino. A instalação é via `cargo install rustlings`, a inicialização é via `rustlings init`, e ele pressupõe Rust/Cargo instalados. ([Rustlings][26]) A documentação também recomenda um editor com `rust-analyzer`, o que aproxima o exercício do fluxo real de desenvolvimento Rust. ([Rustlings][26])

Além disso, Rustlings organiza exercícios por tópicos em diretórios como `variables`, `functions`, `if`, `primitive_types`, `vecs`, `move_semantics`, `errors`, `traits`, `lifetimes`, `tests`, `iterators`, `threads`, `smart_pointers`, `clippy`, `conversions` etc., e mapeia esses tópicos para capítulos do Rust Book. ([GitHub][27]) Isso dá ao aluno uma sensação mais próxima de um curso integrado ao ecossistema real.

A diferença pedagógica central fica assim:

| Dimensão                      | Ruby Koans                                                   | Rustlings                                            |
| ----------------------------- | ------------------------------------------------------------ | ---------------------------------------------------- |
| Unidade principal de trabalho | Método `test_*` com lacunas                                  | Arquivo de exercício Rust com erro/TODO              |
| O que costuma falhar          | Assertion customizada do `Neo`                               | Compilação, testes ou lints                          |
| Ferramenta de feedback        | Runner próprio, mensagens do Sensei                          | `rustc`, `cargo`, testes, `clippy`, watch mode       |
| O que o aluno edita           | Majoritariamente o próprio koan/teste                        | Majoritariamente código que precisa compilar/passsar |
| Realismo de ferramenta        | Baixo a médio                                                | Médio a alto                                         |
| Realismo de projeto           | Baixo na maioria; médio nos projetos                         | Médio; ainda é exercício pequeno, mas usa stack real |
| Tratamento de erro            | Erros aparecem, mas geralmente embalados pelo runner próprio | O erro de compilação é o centro do aprendizado       |
| Filosofia                     | “Descubra o comportamento preenchendo respostas”             | “Conserte código real até compilar/passsar”          |

Rustlings também tem exercícios de teste, inclusive uma pasta `tests`, e a documentação dessa seção diz que muitos exercícios seguintes pedirão para fazer testes passarem. ([GitHub][28]) Mas o ponto é que, mesmo nesses casos, o aluno está lidando com a ferramenta real do ecossistema Rust. Há também uma seção de `clippy`, explicitamente voltada a lints que detectam erros comuns e ajudam a melhorar código Rust. ([GitHub][29])

Ruby Koans, por outro lado, usa uma ferramenta própria que ensina conceitos de Ruby, mas não ensina o fluxo real de um projeto Ruby moderno: `bundle exec`, Minitest/RSpec, `rake test`, `rubocop`, `ruby -w`, `debug`, gems, `Gemfile`, CI, warnings, fixtures, código separado em `lib/` e testes separados em `test/` ou `spec/`.

## Sobre erros de runtime, interpretador e ferramentas

Ruby Koans mostra alguns erros reais de Ruby, mas não transforma isso no eixo central do aprendizado. Por exemplo, `about_methods.rb` testa erro por número errado de argumentos e comenta que isso é erro de runtime, não erro de sintaxe. ([GitHub][11]) `about_classes.rb` usa `eval "fido.@name"` para poder demonstrar uma expressão sintaticamente inválida sem quebrar o arquivo inteiro. ([GitHub][20])

Mas isso é diferente de Rustlings. Em Rustlings, o erro de compilação é frequentemente o exercício. Você lê o diagnóstico do compilador e altera o código. Em Ruby Koans, se você escreve Ruby inválido, pode até ver `SyntaxError` ou outro erro real, mas isso normalmente é acidental ou intermediado pelo runner. A experiência desenhada pelo projeto é: rode, veja o primeiro koan que falhou, medite na mensagem, preencha a lacuna, repita. O runner captura falhas e exceptions dentro dos métodos de koan e para no primeiro problema. ([GitHub][4])

Isso quer dizer que Ruby Koans não é ruim. Ele só é outro tipo de material. Ele é excelente para “eu quero entender comportamento de Ruby por exemplos mínimos”. Ele é fraco para “eu quero sentir como é desenvolver Ruby no dia a dia”.

## Onde o Koans chega perto do Rustlings

Os melhores paralelos com Rustlings são estes:

`triangle.rb` + `about_triangle_project.rb`/`about_triangle_project_2.rb`: o aluno escreve implementação real para satisfazer testes.

`about_scoring_project.rb`: o aluno implementa uma função de score com vários casos.

`about_dice_project.rb`: o aluno implementa uma classe com estado e comportamento.

`about_proxy_object_project.rb`: o aluno implementa delegação dinâmica e tracking de mensagens.

`about_sandwich_code.rb`: há pelo menos um trecho pequeno em que o aluno precisa reescrever uma função usando abstração com bloco.

Esses arquivos são os que mais se aproximam de uma experiência “teste guia, código é alterado”. O problema é que eles são minoria dentro do percurso.

## Minha avaliação final

O Ruby Koans tem profundidade conceitual, mas não tem tanta profundidade prática quanto parece. Ele cobre muitos conceitos da linguagem, só que a maior parte do aprendizado acontece por **preencher respostas em asserts**, não por **escrever código Ruby sob pressão de uma suíte real**.

Eu classificaria assim:

| Critério                                  | Nota qualitativa                                                 |
| ----------------------------------------- | ---------------------------------------------------------------- |
| Aprender semântica básica de Ruby         | Alto                                                             |
| Aprender lendo exemplos pequenos          | Alto                                                             |
| Aprender TDD real                         | Médio-baixo                                                      |
| Aprender Minitest/RSpec                   | Baixo                                                            |
| Aprender tooling Ruby moderno             | Muito baixo                                                      |
| Aprender por erros reais de interpretador | Baixo a médio                                                    |
| Aprender escrevendo implementação         | Médio, por causa dos projetos, mas não pela maioria dos arquivos |
| Realismo comparado ao Rustlings           | Baixo a médio                                                    |

Então, se a pergunta é: “Ruby Koans é tão realista quanto Rustlings?”, minha resposta é **não**.

Se a pergunta é: “Ruby Koans é só corrigir assertions?”, minha resposta é **quase, mas não totalmente**. O coração do projeto é esse, mas há uma camada menor de projetos que realmente pede implementação.

Se a pergunta é: “Daria para criar uma versão melhor inspirada em Rustlings?”, minha resposta é **sim, claramente**. O melhor caminho seria manter o espírito incremental do Koans, mas mudar a mecânica: arquivos em `lib/`, testes reais em `test/` ou `spec/`, runner com `bundle exec`, exercícios que falham por `NameError`, `NoMethodError`, `ArgumentError`, `FrozenError`, `SyntaxError`, warnings, RuboCop, Minitest/RSpec e, em tópicos avançados, RBS/Steep ou Sorbet. Isso ficaria muito mais próximo do que Rustlings faz para Rust.

Assumi que você se refere ao projeto **Edgecase Ruby Koans**. Fiz a análise contra a linha Ruby atual: em 7 de maio de 2026, a lista oficial de releases mostra **Ruby 4.0.3**, publicado em 21 de abril de 2026, e Ruby 4.0.0 publicado em 25 de dezembro de 2025. ([Ruby][1])

Meu veredito é: **o Ruby Koans ainda é bom para fundamentos clássicos de Ruby, mas não está bem alinhado com Ruby 4 como currículo moderno**. Ele ensina muita coisa que continua correta — objetos, classes, módulos, blocos, hashes, arrays, exceptions, `method_missing`, duck typing, etc. — mas está estruturalmente preso a uma visão Ruby 1.8/1.9/2.x, com remendos para Ruby 3. O problema mais grave não é apenas “faltam novidades”: no caminho principal, alguns capítulos modernos são **explicitamente ignorados em Ruby 4** por causa dos gates de versão.

O README ainda diz que qualquer Ruby “maior que 1.8” serve e que o projeto requer “Ruby 1.8.x or later”, o que é um sinal claro de documentação obsoleta para 2026. ([GitHub][2]) O próprio README descreve a proposta como aprender sintaxe, estrutura, funções comuns e bibliotecas por arquivos ordenados em `path_to_enlightenment.rb`, então o caminho de carga importa muito para a avaliação curricular. ([GitHub][2])

O achado crítico: `path_to_enlightenment.rb` carrega `about_keyword_arguments` apenas para versões `"2"` e `"3"`, e carrega `about_pattern_matching` apenas para `"2.7"` e `"3"`. Em Ruby 4, portanto, justamente dois arquivos que deveriam ser centrais para Ruby moderno ficam fora do percurso. ([GitHub][3]) Isso acontece porque o helper `in_ruby_version` usa prefix match em `RUBY_VERSION`, ou seja, `"3"` casa com `3.x`, mas não com `4.x`. ([GitHub][4]) Há outro gate semelhante em `neo.rb`: `method_missing` é tornado público para o koan apenas em `"1.9"`, `"2"` e `"3"`, excluindo Ruby 4. ([GitHub][4])

## O que está atualizado e o que não está

Como material introdutório, eu diria que o projeto está **parcialmente atualizado**: talvez algo como **65% bom para fundamentos**, mas **35% ou menos bom como retrato do Ruby 4 moderno**. A maior parte dos exercícios básicos continua verdadeira, mas a curadoria não reflete a linguagem atual. Ruby 3 e 4 trouxeram mudanças e temas que hoje fazem parte do repertório idiomático: separação rígida de keyword arguments, forwarding com `...`, pattern matching estabilizado, hash value omission, argumentos anônimos `*`/`**`, `it`, `Data`, `Regexp.timeout`/mitigações ReDoS, Prism, RBS/TypeProf, YJIT/ZJIT, Ractor/Fiber Scheduler, Set como core class, RubyGems/Bundler 4, Ruby::Box, e mudanças de string literal/frozen semantics. Vários desses temas aparecem em releases oficiais de Ruby 3.0 a 4.0. ([Ruby][5])

Ruby 4.0, especificamente, introduziu **Ruby Box** e **ZJIT**, melhorias em Ractor, mudanças de linguagem como `*nil` não chamar mais `nil.to_a`, operadores lógicos no começo da linha continuarem a expressão anterior, `Array#rfind`, mudanças em `Binding` para parâmetros implícitos, `instance_variables_to_inspect`, `Pathname` como core class, `Set` como core class e RubyGems/Bundler 4. ([Ruby][6]) Quase nada disso é coberto pelo Koans.

## Lacunas principais por tema

A maior lacuna pedagógica é que **keyword arguments e pattern matching deveriam ser capítulos centrais, não condicionais que somem em Ruby 4**. O projeto até tem arquivos para isso, mas os gates atuais tornam o material invisível na versão mais recente. Isso precisa ser corrigido antes de qualquer outra modernização.

A segunda lacuna é que o projeto quase não cobre Ruby 3.x como “Ruby moderno”. Ruby 3.0 separou keyword arguments de argumentos posicionais e estabilizou pattern matching; Ruby 3.1 adicionou omissão de valores em hash/keywords; Ruby 3.2 trouxe argumentos anônimos `*`/`**`, `Data` e melhorias de regex/ReDoS; Ruby 3.3 trouxe Prism; Ruby 3.4 trouxe `it`, mudanças em string literals e modular GC. ([Ruby][5])

A terceira lacuna é idiomática. Alguns exercícios ainda ensinam formas válidas, mas não as mais representativas: hash rockets em vez de `{key: value}`, `send` sem discutir `public_send`, proxy com `method_missing` sem `respond_to_missing?`, `open` em vez de `File.open`, `collect`/`inject` sem equilibrar com `map`/`reduce`/`each_with_object`, e pouco tratamento de keyword args.

## O que eu mudaria para deixar Ruby Koans alinhado a Ruby 4

Primeiro, corrigiria todos os gates de versão. Uma mudança mínima seria substituir gates como `in_ruby_version("2", "3")` por algo baseado em `Gem::Version`, por exemplo `ruby_version_at_least?("2.7")` ou `ruby_version_at_least?("3.0")`. Isso resolveria o problema estrutural de Ruby 4 pular capítulos.

Segundo, eu adicionaria um bloco novo chamado algo como `about_modern_ruby.rb`, ou dividiria em arquivos menores:

`about_keyword_arguments_modern.rb`: separação Ruby 3+, `**kwargs`, `**nil`, `...`, forwarding com leading args, `ruby2_keywords`, `public_send` com kwargs.

`about_pattern_matching_modern.rb`: `case/in`, one-line matching, find pattern, pin com expressão, `deconstruct`, `deconstruct_keys`, `Data`, matching em objetos de domínio.

`about_blocks_modern.rb`: `_1`, `_2`, `it`, lambdas versus procs, forwarding de blocos e argumentos anônimos.

`about_data_and_value_objects.rb`: `Struct`, `Data`, imutabilidade, `with`, equality, pattern matching.

`about_regex_modern.rb`: `match?`, named captures, `Regexp.timeout`, ReDoS, `MatchData#byteoffset`.

`about_tooling.rb`: Bundler, Gemfile, `bundle exec`, Minitest/RSpec, debug.gem, error_highlight, syntax_suggest, Prism.

`about_concurrency.rb`: Fiber Scheduler, Ractor, shareability, `Ractor::Port`, limites práticos.

`about_runtime.rb`: YJIT, ZJIT, RubyGems/Bundler 4, default/bundled gems, Set/Pathname como core classes.

Terceiro, eu revisaria textos enganadores ou historicamente marcados. `test_hash_is_unordered` deveria virar algo como `test_hash_equality_ignores_order`, com outro teste mostrando que `keys` preserva ordem de inserção. O comentário de symbols deveria explicar a história de symbol GC. `about_sandwich_code` deveria trocar `open` por `File.open`. `about_message_passing` e `about_proxy_object_project` deveriam usar `respond_to_missing?` e tratar keyword forwarding.

## Conclusão

O Ruby Koans da Edgecase **não está quebrado como material de fundamentos**, mas **está obsoleto como trilha de Ruby 4**. Ele ainda ensina conceitos que todo rubyist precisa saber, só que deixa de ensinar uma parte grande do Ruby que define código idiomático de 2020–2026. O problema mais sério é mecânico: no Ruby 4, capítulos sobre keyword arguments e pattern matching são pulados pelo próprio `path_to_enlightenment.rb`. Depois vêm lacunas curriculares: `Data`, `it`, forwarding moderno, `Regexp.timeout`, Prism, Ractor/Fiber, YJIT/ZJIT, Ruby::Box, Set/Pathname core e RubyGems/Bundler 4.

Minha recomendação técnica seria não descartar o projeto, mas tratá-lo como **Ruby Koans clássico**. Para ensino em 2026, ele precisa de uma camada “Ruby 3/4 Koans” e de correções de compatibilidade para Ruby 4 antes de ser considerado atualizado.
